# ComplianceAgent — Model Governance & Monitoring

> Companion to the [Model Validation Report](MODEL_VALIDATION.md). Defines *who owns
> what*, the **operating-point policy**, the **ongoing-monitoring plan**, and the
> **model-risk register** — the governance a regulated deployment requires (structured
> after SR 11-7 §VI, "Governance, Policies, and Controls"). Portfolio/synthetic system;
> roles below are illustrative of the target operating model.

## 1. Roles (RACI)

| Activity | Model Owner | Developer | Independent Validator | MLRO / Approver |
|---|:--:|:--:|:--:|:--:|
| Model development | A | R | C | I |
| Independent validation | I | C | R/A | I |
| Operating-point (threshold) sign-off | R | C | C | **A** |
| Production deployment | R | C | C | **A** |
| Ongoing monitoring | **A/R** | C | C | I |
| Incident / drift response | A | R | C | I |

*A=Accountable, R=Responsible, C=Consulted, I=Informed.*

## 2. Operating-point (threshold) policy

The GNN emits **calibrated probabilities**; the account-flagging **threshold is a
governed control**, not a hardcoded 0.5. It is set from an operating-point analysis
(`eval/operating_point.py`) against a documented **recall-vs-alert-volume** target and
signed off by the MLRO. On the real IBM benchmark (base rate ~1.2%):

| Target recall | Threshold | Precision | Alert (flag) rate |
|--:|--:|--:|--:|
| 70% | 0.101 | 5.3% | 16.3% |
| 80% | 0.081 | 4.6% | 21.7% |
| **90% (recommended)** | **0.057** | 3.5% | 31.4% |
| 95% | 0.033 | 2.6% | 45.5% |

**Policy:** default target **≥90% recall** (regulators penalise missed SARs far more
than false positives, which analysts triage). The chosen threshold is set via
`GNN_FLAG_THRESHOLD` and surfaced per case as `flagged_accounts` + `flag_threshold`.
A deployment **re-derives** this on its own labeled data before go-live.

## 3. Ongoing-monitoring plan

Automated, with action thresholds. Metrics are exposed at `GET /api/model` and
regenerated by the eval suite.

| Metric | Source | Green | Action trigger |
|---|---|---|---|
| **Unsupported-claim rate** | `eval/hallucination.py` | 0% | > 1% → freeze LLM narratives, investigate |
| **NLI faithfulness (mean)** | Verifier | ≥ 0.70 | < 0.60 sustained → prompt/model review |
| **Verifier failure / retry rate** | Audit log | low | spike → model or data-quality review |
| **GNN input drift (PSI)** | `gnn/drift.py` | < 0.1 | 0.1–0.25 monitor · > 0.25 **re-train** |
| **Calibration (ECE)** | model metrics | < 0.10 | > 0.15 → re-calibrate |
| **Retrieval Recall@5** | `eval/retrieval_eval.py` | ≥ 0.90 | < 0.85 → KB / retriever review |
| **Sanctions feed freshness** | `/api/admin/sanctions` | < 30 days | stale → refresh cron check |

**Cadence:** guardrail metrics per-run (live); drift + calibration weekly;
full validation quarterly; independent review annually (or on material change).

## 4. Model-risk register

| # | Risk | Likelihood | Impact | Mitigation | Residual |
|--:|---|:--:|:--:|---|:--:|
| R1 | LLM fabricates an unsupported claim | Low | High | Deterministic draft + claim/citation/figure checks + NLI + abstention; measured 0% | **Low** |
| R2 | Over-confident (mis-calibrated) risk score | Med | Med | Platt calibration; ECE monitored; ensemble + human gate | Low |
| R3 | Missed true positive (low recall) | Med | High | Operating point tuned to ≥90% recall; sanctions hard-override; human review | Med |
| R4 | Population drift degrades the GNN | Med | Med | PSI drift monitor with re-train trigger | Low |
| R5 | Prompt injection via case data | Low | High | Input guardrails + 11-attack red-team in CI | Low |
| R6 | Stale sanctions/PEP data | Low | High | Scheduled OpenSanctions/OFAC/UN refresh + freshness monitor | Low |
| R7 | Autonomous action without human | — | Critical | **Structurally impossible** — approval gate enforced in code | **None** |

## 5. Change management

- **Versioned model registry** + model cards; every model has a reproducible training command.
- **Re-training triggers:** drift > 0.25 PSI, calibration ECE > 0.15, or a material data/regulatory change.
- **Promotion:** validator review → MLRO sign-off → deploy; the prior version is retained for rollback.
- **Prompt governance:** prompts are versioned (`PROMPT_VERSION`) and peer-reviewed before a bump.
